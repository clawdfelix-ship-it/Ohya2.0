# 參考 macrozheng/mall：整體改造藍圖（Roadmap）

## 背景
目標係將現有 Node/Express + EJS + Postgres 嘅電商系統，逐步升級到「可營運、可分工、可擴展」嘅後台能力；參考 `macrozheng/mall` 嘅產品能力與運作方式（PMS/OMS/UMS 思路），但不直接搬 Spring/Vue 技術棧。

參考來源（能力/思路）：
- `macrozheng/mall` 專案介紹（PMS/OMS/UMS 模組視角）
- `mall-learning` 權限模型：菜單/資源/角色/後台用戶（UMS 思路）

## 設計原則（跟 mall 對齊但符合現況）
- **先營運後完美**：先把「每日會用到」嘅後台流程做好，再做架構優化。
- **資料模型先行**：功能擴展以 DB schema 約束 + API 校驗確保一致性。
- **可分工（RBAC）必做**：後台不再靠「全 admin 一刀切」。
- **兩級分類先跑通**：先穩定兩級（大分類＞子分類），避免一開始引入多層/屬性系統爆炸。
- **演進式對照**：每個 phase 都有「對照 mall：我們補咗咩、仲差咩」。

## 現況快照（系統已有）
- 後台頁面：商品/分類/訂單/用戶/售後(returns)/退款(refunds)/對賬(reconciliation)。
- DB：`categories.parent_id` 已可表達層級；`products.category_id` 目前要求 leaf。
- 權限：目前只有 `is_admin` gating（無角色/菜單/資源權限）。

## Roadmap（B → A → C）

### Phase 1（PMS 優先）：分類樹 + 商品中心（最先提升日常營運）
#### 1.1 分類管理分層（大分類＞子分類）
- 後台 `/admin/categories`：
  - 左側列表以樹狀縮排呈現兩級，可展開/收合。
  - 大分類行支援「新增子分類」快捷操作（帶入 parent_id）。
  - 右側表單新增「上級分類」欄位（parent_id）。
  - 防呆：只允許兩級、禁止自指/循環、避免三層。
- Spec 參考：`2026-05-04-admin-categories-hierarchy-design.md`

#### 1.2 商品中心補強（對齊 mall 的 PMS 必要能力）
最少落地項目（不一次做到 mall 全量）：
- SKU/成本/庫存（已部分存在，補齊後台操作閉環）
  - SKU 列表/編輯（包含成本、售價、啟用、庫存調整入口）
  - 庫存調整要落流水（inventory transactions），可追溯
- 商品屬性（輕量版）
  - 先做「商品標籤/品牌」管理頁（如現有 API 已有但缺 UI）
  - 先不做 mall 式「屬性分類/規格模板」全家桶，避免 scope 爆炸

#### 1.3 分類/商品 UI 一致化
- 後台列表要支援：搜尋、排序、快速篩選（狀態/分類）
- 統一表單/錯誤顯示/確認刪除等交互規則

**Phase 1 驗收標準**
- 分類管理可以穩定維護兩級樹；新增子分類效率高；不會產生第三層/循環。
- 商品編輯可以穩定揀子分類（leaf），並可見 SKU/成本/庫存基本資料。

---

### Phase 2（UMS/RBAC）：真正可分工的後台（參考 mall-learning）
目標係由「admin 一刀切」升級到：
- 角色（Role）：例如 營運/客服/財務/倉務/超級管理員
- 菜單（Menu）：控制後台左側/頂部導航可見性（頁面級）
- 資源（Resource）：控制 API 訪問（路徑匹配 / 方法級）
- 用戶（Admin User）：可分配多角色

落地順序（由易到難）：
1) 角色管理 + 用戶分配角色（先做到「誰可以入後台哪些頁」）
2) 菜單管理（先固定一套菜單定義，後台可配置顯示/排序）
3) 資源管理（API 權限），逐步把敏感操作收口（退款 approve、對賬導出、物流 webhook 手動補單等）

**Phase 2 驗收標準**
- 可新增角色，並把現有後台用戶分配角色後，菜單顯示會變。
- 沒有資源權限的角色無法呼叫受保護 API（返回 403）。

---

### Phase 3（OMS/財務）：訂單/售後/退款/對賬做深（對齊 mall 交易閉環）
目標係將現有 orders/refunds/returns/reconciliation 提升到可營運水準：
- 訂單狀態（履約）/支付狀態/售後狀態清晰拆分
- 退款工作流（部分/全額/多次退款）+ 退款對賬導出
- 對賬報表（daily → 可下載 CSV、可按支付渠道/日期區間）
- 履約補強（可選）：出貨批次、包裹、物流面單/狀態同步（視 ShipAny 落地深度）

**Phase 3 驗收標準**
- 財務可按日/渠道拉對賬報表；退款單可追溯到支付流水；狀態不混亂。

## 非目標（避免變成「重寫 mall」）
- 不把技術棧改成 SpringBoot/Vue。
- 不一次性引入 Elastic/RabbitMQ/微服務拆分（除非後續營運規模需要）。
- 不先做三層分類/完整屬性系統（Phase 1 先跑通兩級 + SKU/庫存閉環）。

## 風險與控制
- 外部資料品質（分類來源 MZAKKA breadcrumb 不穩定）：以「兩級分類」+ fallback 規則控制；後台允許人工修正。
- 權限導入風險（鎖死自己）：保留超級管理員 bypass + emergency fallback（僅本地/配置允許）。
- 既有資料遷移：每個 phase 都提供可重跑的 backfill script + 事前驗證查詢。

## 交付形式
- 每個 phase：Spec → Plan → Implementation → 驗收清單
- 每次改動保留自動化測試覆蓋（node:test）
